# AI Output Is Untrusted Code

**Plain language:** Your AI's code is just another stranger's code — review it before you run it.

## What it is

Code generated by an agent is not the same kind of object as code written by a known author. You don't know what context the model had when it wrote it. You don't know whether a hostile webpage in the agent's retrieval set nudged the output. You don't know whether the agent confused two function signatures and produced a plausible-looking but wrong call.

So `ai-output-is-untrusted-code` is the operational corollary of [`untrusted-by-default`](contextqb://principles/untrusted-by-default): the agent itself is an external source. Code it generates that **runs** — tool calls, file edits, shell commands, API requests, database mutations — crosses into your system from outside. Treat it the way you'd treat code from an anonymous pull request, not the way you'd treat your own code.

This does not mean "don't use agents." It means: build the boundary between "agent proposed" and "system executed" deliberately, and keep that boundary visible.

## Why it matters in agentic dev specifically

Three forces converge to make this principle load-bearing:

1. **Auto-execute is the default.** Most agent IDEs and MCP clients are configured to execute tool calls automatically. The agent suggests a shell command and runs it in the same breath. This is the convenient default and the most expensive one — there is no review window where a human or policy can intervene.

2. **Capability accretion is silent.** Yesterday the agent could only read files. Today you wired up shell access for a one-off task. Tomorrow you forgot to remove it. The agent's blast radius has grown without anyone tracking it.

3. **Trust transfers by mistake.** A user who has watched an agent succeed ten times in a row starts approving the eleventh without reading it. This is a well-studied human pattern; it's also exactly the moment something destructive ships.

The combined effect is that agentic systems make it trivial to execute untrusted code at the speed of conversation. Without a deliberate boundary, every prompt is a `curl | bash` against your codebase.

## Minimum acceptable posture

You can claim this principle if all of the following hold:

1. **No auto-execute for destructive operations.** Anything that deletes, deploys, sends, charges, or mutates external state requires explicit human (or automated policy) approval. "Read this file" can auto-execute; "remove this file" cannot.

2. **Generated code that mutates external state is reviewed before merge.** Even if the agent wrote it, a human or a CI gate reads it before it lands in main. Agent-generated diffs are not exempt from review.

3. **Secrets are not in the agent's reach during normal operation.** The agent does not have ambient access to `.env`, the production database, or your cloud account. If it needs them for a specific task, the access is scoped, time-bound, and named.

4. **The agent's tool list is enumerable.** You can name every tool the agent can call and what each one does. "Whatever the IDE supports" is not an answer.

5. **A failed agent action is logged.** If the agent tried to do something it wasn't allowed to do, there is a record. You learn from refusals, not just from successes.

## Signals you're getting this wrong

- **You've stopped reading the diffs.** The agent ships PRs faster than you read them, and "looks fine" has become the rubric.

- **Shell access is "for convenience."** No one can articulate which specific commands the agent actually needs.

- **The same secret is in three places.** The agent saw it once, copied it into a comment, mentioned it in a generated log line — and now it's everywhere.

- **The agent has been "fixing" production.** Production incidents involving agent commits exceed production incidents involving human commits.

- **You can't reproduce what the agent did.** A change happened, no one can find the prompt, no one can find the diff, and the tests still pass.

## How it relates to other ContextQB principles

**Untrusted by Default** — That principle establishes the mental model: external input is hostile until validated. This principle is its operational corollary: the agent is external. Validate its output the same way.

**Failure Modes** — Agent-generated code has its own failure modes. The most distinctive: confidently wrong code. It will never look broken; it will only behave broken. Anticipate that.

**State Ownership** — Generated code that mutates state is a state ownership question. Who owns the result if the agent wrote it? The user who prompted it owns the consequences. That ownership has to be visible in code, not just in the chat log.
